**ASSISTANT:** That's a clean design. You've separated two concerns that don't need to be bundled together:

- **Data transfer** → temp file (let the OS handle it, GeoPandas `.to_file()` does the work)
- **Control signal** → tiny socket message (just paths, no bulk data)

The socket never carries data, only a pointer to data. The message is:

```json
{
  "layer_name": "Risk Zones",
  "file_path": "/tmp/qbridge_abc123/layer.geojson",
  "qml_path": "/tmp/qbridge_abc123/style.qml",
  "layer_type": "vector",
  "update_existing": true
}
```

QGIS receives that, calls `iface.addVectorLayer(file_path, layer_name, "ogr")`, then applies the QML if provided. That's maybe 10 lines in the plugin.

**Why this is smart:**
- File I/O to `/tmp` is fast — well under 100ms for anything a human would visually inspect
- QML is QGIS's native style format — full styling fidelity, no re-implementation of QGIS's renderer API
- The QGIS plugin stays trivial — no data parsing, all format handling done by QGIS's OGR provider

## Key Decisions Made This Session

| Decision | Rationale |
|---|---|
| Transport: local TCP socket | Real-time, stdlib only, no polling, platform-consistent |
| Data via temp file, not socket | Keeps socket message tiny; GeoPandas/OGR handle serialization natively |
| Styling via QML file | QGIS native format; full fidelity; no re-implementation of renderer API |
| Plugin in same repo | Single source of truth; easier to keep protocol in sync; easier to contribute |
| Port 45678 | Unlikely to conflict with common services |
| Newline-delimited JSON | Simplest framing; easy to parse with stdlib |
| One message → one response per call | Stateless; simple to reason about; no connection management |
